## Context

活動建立表單（`EventPage.vue`）目前用單一 `deadline` 欄位＋固定預設偏移量算截止時間，四個情境（A 固定時段、B 時間窗投票、C 候選日投票、D 候選日+時段投票）共用同一套 `scheduleAnchor`／`isUrgent`／`deadlineDisplayText` 計算邏輯，但這套邏輯有結構性缺陷：預設值選取沒有安全緩衝、緊急判斷依據錯誤訊號、情境間文案不一致。同時 `ActivityDetailModal.vue`（決選畫面）跟 `AvailabilityPickerModal.vue`（報名者回報可用時間）都沒有處理候選時段/候選日期已經過期的狀態。

這份 change 是「截止時間邏輯統一重新設計」計畫的前端部分，後端 `BuJoBackend` repo 有平行進行的對應 change 調整 `deadline_at`/`vote_deadline_at` 的 API 語意，兩者透過既有 API contract（`deadline_at`、`vote_deadline_at` 欄位）銜接，不直接耦合實作進度。

## Goals / Non-Goals

**Goals:**

- 建立表單的截止時間顯示要讓使用者一次看清楚兩件不同的事：報名什麼時候截止（可調整）、活動最晚什麼時候會被系統自動取消（不可調整）
- 自動選取的預設值必須保證有足夠的安全緩衝，不能出現「技術上還沒過去、但實際上只剩幾分鐘」的默默失敗
- 緊急狀態的判斷依據要跟「算出來的截止時間本身是否貼近現在」直接掛鉤，不能依賴「活動本身距今多久」這種間接訊號
- 決選畫面跟報名者的時間選取畫面，都要讓使用者一眼看出哪些候選項目已經過期、不能再選

**Non-Goals:**

- 不處理任何通知功能（新增通知型別、通知呈現/已讀狀態）——這是另一位組員的工作範圍
- 不修改後端 API 呼叫的資料結構或新增/移除欄位——後端語意調整在 `BuJoBackend` repo 的平行 change 處理，本次只消費既有的 `deadline_at`/`vote_deadline_at`/`candidate_slots`/`decision_candidates` 回應形狀
- 不處理「報名截止後隱藏活動卡片給非報名者/非建立者」的存取限制——獨立的另一個 change

## Decisions

### 智慧預設演算法：固定目標值 + 單向降級，取代「永遠選最大有效偏移量」

**決策**：預設值選取不再是「由大到小依序找還沒過去的第一個偏移量」，改成固定以「3 小時前」為一般情況的目標值；只有當「3 小時前」算出來的報名截止時間距現在不足 30 分鐘緩衝時，才依序降級試「1 小時前」→「30 分鐘前」→ 都不安全時降級成「無報名緩衝」（報名截止時間等於決策硬截止時間本身）。「12 小時前」「1 天前」這兩個較大的偏移量只保留給使用者手動選擇，演算法不會自動選到。

**理由**：舊演算法「永遠選最大有效偏移量」在活動距今較近（例如明天）時，容易選中一個技術上「還沒過去」但實際緩衝只剩幾十分鐘的偏移量，且完全沒有警示（因為緊急判斷檢查的是另一個訊號）。改成固定目標值後，只要活動時間夠遠，任何情況都會得到一致、可預期的 3 小時報名緩衝，不會因為活動剛好落在某個偏移量的邊界而產生落差極大的結果。

**替代方案考慮**：曾考慮依「活動是否為當天」分兩套邏輯，但「是否同一天」是比「距離現在實際還有多久」更粗糙的判斷依據（同一天內可能是 30 分鐘後也可能是 23 小時後），單向降級演算法已經能透過直接比較時間差自然涵蓋所有距離遠近的狀況，不需要額外分支。

### 兩個獨立時間點的常駐顯示，取代「調整列 vs 警示框」互斥關係

**決策**：建立表單底部固定顯示兩行文字：第一行顯示報名截止時間（可調整，點擊偏移量數字觸發預設編輯器），第二行顯示決策硬截止時間（固定值，系統自動取消活動的最後期限，不可調整）。兩行各自獨立判斷「距今是否 ≤30 分鐘」，命中時該行單獨轉為警示樣式，不需要另外彈出警示框或讓其中一行整段消失。

**理由**：現行版本的「調整列」跟「警示框」是互斥的（緊急時調整列整個消失、只剩警示框），且警示框的文案語意上其實在講「決策硬截止時間」，卻掛在原本用來調整「報名截止時間」的位置，兩個不同語意的時間點被混在同一個 UI 元件裡，使用者無法同時理解兩者的關係。拆成兩個獨立、常駐顯示的行，才能讓使用者一次看清楚「我還能報名多久」跟「我最晚什麼時候要手動確認」是兩件不同的事。

### `isUrgent` 判斷依據改為「算出來的截止時間距今多久」

**決策**：緊急狀態判斷不再檢查「活動本身（`scheduleAnchor`）距今是否 ≤1 小時」，改成分別檢查「報名截止時間距今是否 ≤30 分鐘」與「決策硬截止時間距今是否 ≤30 分鐘」。

**理由**：活動本身的時間跟算出來的截止時間，在活動距今很近時會同步觸發，但活動距今稍遠時（例如明天）就會脫鉤——這時候即使自動選中的偏移量算出一個貼近現在的截止時間，舊邏輯也偵測不到，因為它看的是錯誤的訊號。改成直接檢查「算出來的結果本身」，才能涵蓋這個現有邏輯完全偵測不到的情況。

### 二次確認 modal 觸發範圍縮小到「連降級 fallback 都救不了」的極端情況

**決策**：`showUrgentConfirm` modal 保留，但只有「決策硬截止時間本身距今 ≤30 分鐘」（代表智慧預設演算法的五層 fallback 全部無法提供安全緩衝）才跳出，不再是現行「活動距今 ≤1 小時」這種寬鬆條件。

**理由**：決策硬截止時間到期沒有手動確認會讓整個活動直接消失，這種有實質後果的操作值得一個明確的確認關卡；但既然智慧演算法已經把「報名緩衝過短卻無警示」的情況大幅減少，這個 modal 的觸發頻率也應該同步降低，只保留給真正沒有任何緩衝可言的邊緣案例，避免像現行版本一樣頻繁打斷建立流程。

### 情境二時間窗從「選填」改為「必填」

**決策**：`EventPage.vue` 情境二（`dateMode: 'fixed'`, `timeMode: 'vote'`）的時間窗開始/結束時間，從現行「選填、留空時後端預設全天 00:00–23:59」改成必填，比照情境一 `timeError` 的即時驗證模式。

**理由**：新的截止時間模型下，情境二的決策硬截止時間錨點是「`fixed_date` + 時間窗開始時間」——如果時間窗留空，這個錨點就沒有明確依據，只能退化成一個不一定合理的預設值（例如當天 00:00，對當天建立的活動而言直接等於已經過去）。要求時間窗必填，才能保證這個錨點在任何情況下都有意義。

### 決選候選清單與 `AvailabilityPickerModal` 的過期狀態呈現：維持顯示＋停用樣式＋非顏色標示

**決策**：已過期的候選項目（決選清單裡的候選時段、`AvailabilityPickerModal` 裡的候選日期格子）維持顯示在原本的位置，不整個移除；套用停用樣式（降低透明度、游標改為不可互動、拿掉點擊/hover 行為）；額外加上非顏色的標示（決選清單用文字標籤如「已過期」，`AvailabilityPickerModal` 的日曆格子放不下文字標籤，改用數字加刪除線或角落小圖示）。

**理由**：整個移除會讓使用者困惑「這個選項怎麼不見了」，也會漏失原本已經存在的參考資訊（決選清單的票數/頭像）；只靠灰階顏色沒有額外標示，色弱使用者或匆忙瀏覽時容易漏看；維持顯示但點擊才報錯，是「事後報錯」而非「事前防止」，會讓使用者先花力氣互動才被告知不能選。停用樣式＋非顏色標示同時滿足「保留參考資訊」「事前就看得出不能選」「不只靠顏色傳達語意」三個要求。

**適用範圍差異**：決選清單的過期判斷適用情境二／三／四（情境一沒有候選時段，不會進入決選清單邏輯）；三個情境的候選資料形狀不同，情境三／四是 `candidate_slots`（用 `slot_start` 判斷），情境二是後端 `computeRangeRanking` 算出的 segments（沒有 `slot_start` 這個欄位，要用該 segment 自己的時間範圍起點判斷）。`AvailabilityPickerModal` 的過期判斷則依 `dateWindows[key].end`（情境四逐日窗口）／全域 `timeWindowStart`/`timeWindowEnd`（情境二/三）／當天 23:59（三者都沒有時的預設）為準。

## Implementation Contract

### 行為：建立表單底部的截止時間顯示

- **正常狀態**（報名截止時間距今 >30 分鐘）：第一行顯示「報名開放到 {報名截止時間}（{偏移量文字}截止）」，偏移量文字（例如「3 小時前」）本身可點擊，點擊後展開既有的預設選項清單（1 天前／12 小時前／3 小時前／1 小時前／30 分鐘前），選其他選項後即時重新計算並更新顯示
- **警示狀態**（報名截止時間距今 ≤30 分鐘）：第一行文字改為「報名開放到 {報名截止時間}——活動快開始了，已經沒有緩衝時間」，整行套用警示視覺樣式（例如強調色文字＋圖示）
- 第二行不受使用者互動影響，恆為固定值：正常狀態顯示「最晚 {決策硬截止時間} 要手動確認成團，不然活動會自動取消」，警示狀態（距今 ≤30 分鐘）顯示「只剩 {N} 分鐘了，記得手動確認成團，不然活動會被自動取消喔」，同樣套用警示樣式
- 第三行只在情境三／四、且使用者已選的候選日期中有任一天距今 ≤1 小時時才渲染，內容為候選日提醒文字（情境三／四使用同一份文案），不影響第一/二行的計算或渲染

### 行為：智慧預設演算法的計算結果

給定決策硬截止時間 `deadline_at`（依情境已由天花板規則決定，見計畫文件）與目前時間 `now`：

1. 計算 `candidate = deadline_at - 3小時`；若 `candidate - now >= 30分鐘`，採用 `candidate` 作為報名截止時間，偏移量標記為「3 小時前」
2. 否則計算 `candidate = deadline_at - 1小時`；若 `candidate - now >= 30分鐘`，採用該值，偏移量標記為「1 小時前」
3. 否則計算 `candidate = deadline_at - 30分鐘`；若 `candidate - now >= 30分鐘`，採用該值，偏移量標記為「30 分鐘前」
4. 否則採用 `candidate = deadline_at`（報名截止時間等於決策硬截止時間，即無報名緩衝），偏移量顯示為警示文案（見上方「警示狀態」）
5. 使用者透過點擊偏移量文字手動選擇「12 小時前」或「1 天前」時，不受此演算法限制，直接套用使用者選擇的固定偏移量（即使算出來的報名截止時間因此落在安全緩衝以內，也視為使用者已知情的手動選擇，不強制降級，但仍套用警示樣式提示）

### 行為：二次確認 modal

- 送出表單時，若「決策硬截止時間距今 ≤30 分鐘」，攔截送出並顯示確認 modal，文案為「活動即將開始，這次建立將不會有任何報名緩衝時間，送出後請立即到活動頁面手動確認成團」，使用者需點擊確認按鈕才會實際送出
- 若「決策硬截止時間距今 >30 分鐘」，即使報名截止時間本身觸發警示樣式，送出時也不攔截，直接正常送出

### 行為：情境二時間窗必填驗證

- 送出前驗證：情境二（`dateMode: 'fixed'`, `timeMode: 'vote'`）若時間窗開始或結束時間任一為空，顯示即時錯誤提示並阻止送出，錯誤呈現方式與情境一 `timeError`（例如「請選擇開始時間」的欄位下方即時提示＋捲動至該欄位）一致

### 行為：`ActivityDetailModal.vue` 的 `actionError` 重置

- 元件的 `fetchActivity(activityId)` 函式在開始抓取新活動資料時，須將 `actionError` 重置為空字串，跟現有已經會重置的其他 UI 狀態（候選 chip 展開狀態、頭像彈出框等）走同一個重置時機，確保切換到不同活動時不會殘留舊活動的報名錯誤訊息

### 行為：決選候選清單的過期項目呈現

- 決選候選清單（情境二／三／四適用）的每一筆候選項目，渲染前先判斷是否已過期（時間起點早於目前時間）；已過期的項目維持渲染在清單中原本的排序位置，套用停用樣式（不接受點擊/hover 互動、視覺降低對比度），並顯示文字標籤（例如「已過期」）
- 情境三／四的過期判斷依據該候選項目的 `slot_start` 欄位；情境二的過期判斷依據後端回傳的 segment 自身時間範圍起點（欄位名稱以後端實際回應為準，不是 `slot_start`）

### 行為：`AvailabilityPickerModal.vue` 的日期格子過期呈現

- 日曆渲染每個日期格子時，除了現有的「是否在候選範圍內」判斷，新增「該日期對應的候選窗口是否已完全過去」判斷：依序檢查 `dateWindows[該日期].end`（情境四逐日窗口存在時）、全域 `timeWindowStart`/`timeWindowEnd`（存在時取兩者中較晚者）、或當天 23:59（前兩者都不存在時的預設邊界）
- 已過期的日期格子不可點擊（點擊事件不觸發任何選取行為）、套用停用視覺樣式，並額外附加非顏色標示（數字加刪除線，或格子角落疊加小圖示），不能只靠灰階顏色區分

### Scope 邊界

- **In scope**：`EventPage.vue` 的截止時間顯示/計算/驗證邏輯；`ActivityDetailModal.vue` 的 `actionError` 重置與決選候選清單過期呈現；`AvailabilityPickerModal.vue` 的候選日期格子過期呈現
- **Out of scope**：任何通知型別的新增或呈現；後端 API 的資料結構異動；`ActivityView.vue`／`DateEventsModal.vue` 等其他消費 `ActivityDetailModal.vue` 的容器元件（本次只修元件本身，容器已有的重用模式不受影響）

## Risks / Trade-offs

- **[Risk]** 決選清單情境二（`computeRangeRanking` segments）跟情境三/四（`candidate_slots`）的過期判斷欄位形狀不同，實作時若沒有正確區分容易誤用欄位名稱 → **Mitigation**：在共用的過期判斷 helper 裡明確依 `scheduleVariant` 分支取值，並在測試中為兩種資料形狀分別涵蓋案例
- **[Risk]** 智慧預設演算法的降級/警示門檻（30 分鐘）目前是單一固定常數，若未來想調整緩衝門檻，需要同時改動演算法跟 modal 的觸發條件（兩者共用同一個 30 分鐘門檻）→ **Mitigation**：實作時把這個門檻抽成單一命名常數共用，不要在兩處各自硬編碼
- **[Risk]** `AvailabilityPickerModal.vue` 的日期格子目前沒有渲染文字標籤的空間，若「刪除線/圖示」的視覺方案在小螢幕上不夠清楚，可能仍然不符合「不能只靠顏色」的要求 → **Mitigation**：實作後在小尺寸裝置（375px 寬）手動驗證標示的可辨識度
